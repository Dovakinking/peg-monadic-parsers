# Формальная часть проекта: PEG как монадические парсеры на Haskell

## 1. Постановка задачи

**Вход:** файл с описанием PEG-грамматики `G` и строка `w`.

**Выход:** ответ, принадлежит ли `w` языку `L(G)`. Строка считается принадлежащей языку, если стартовое правило успешно разбирает всю строку, то есть после разбора позиция равна длине входа.

Проект также умеет генерировать Haskell-модуль, где каждое правило PEG является определением монадического парсера типа `Parser ()`.

## 2. Формат входной грамматики

Используется подмножество синтаксиса Peggy.js без семантических действий и JavaScript-предикатов:

```ebnf
Grammar     ::= Rule+
Rule        ::= Identifier "=" Expression ";"?
Expression  ::= Sequence ("/" Sequence)*
Sequence    ::= Prefix*
Prefix      ::= "&" Prefix | "!" Prefix | Suffix
Suffix      ::= Primary ("*" | "+" | "?")*
Primary     ::= Identifier
              | StringLiteral
              | CharClass
              | "."
              | "(" Expression ")"
```

Терминалы: строковые литералы в одинарных или двойных кавычках, символьные классы вида `[a-z]` и `[^0-9]`, точка `.` для любого одного символа. Пустое выражение можно записать как `""`, `eps`, `epsilon` или `ε`.

Первое правило является стартовым, если в CLI не указан параметр `--start`.

## 3. Абстрактный синтаксис PEG

После разбора файл переводится в AST:

```haskell
data Expr
  = Epsilon
  | Literal String
  | AnyChar
  | CharClass Bool [ClassAtom]
  | NonTerminal Name
  | Sequence [Expr]
  | Choice [Expr]
  | And Expr
  | Not Expr
  | Many Expr
  | Some Expr
  | Optional Expr
```

`Choice` является **упорядоченным** выбором, а не симметричной альтернативой: если первая ветка успешно разобрала префикс, последующие ветки не рассматриваются.

## 4. Денотационная семантика

Пусть `w` - входная строка длины `n`, а `i` - текущая позиция, `0 <= i <= n`. Для выражения `e` определим частичную функцию:

```text
[[e]]_G(w, i) ∈ {Fail} ∪ {Ok j | i <= j <= n}
```

`Ok j` означает, что `e` успешно разобрало префикс `w[i..j)` и оставило позицию `j`. `Fail` означает отказ.

Основные правила:

```text
[[ε]](w, i) = Ok i

[["s"]](w, i) = Ok (i + |s|), если s является префиксом w[i..]
              = Fail, иначе

[[.]](w, i) = Ok (i + 1), если i < |w|
            = Fail, иначе

[[A]]_G(w, i) = [[R(A)]]_G(w, i), где R(A) - правая часть правила A
```

Последовательность:

```text
[[e1 e2]](w, i) =
  если [[e1]](w, i) = Ok j, то [[e2]](w, j),
  иначе Fail.
```

Упорядоченный выбор:

```text
[[e1 / e2]](w, i) =
  [[e1]](w, i), если [[e1]](w, i) = Ok j;
  [[e2]](w, i), если [[e1]](w, i) = Fail.
```

Позитивный и негативный просмотр вперед:

```text
[[&e]](w, i) = Ok i, если [[e]](w, i) успешно;
             = Fail, иначе.

[[!e]](w, i) = Ok i, если [[e]](w, i) = Fail;
             = Fail, иначе.
```

Повторы:

```text
[[e*]](w, i) = результат жадного повторения e, пока e успешно потребляет хотя бы один символ.
[[e+]](w, i) = [[e e*]](w, i).
[[e?]](w, i) = [[e / ε]](w, i).
```

Если `e` может успешно завершиться без продвижения позиции, выражения `e*` и `e+` статически запрещаются валидатором: иначе greedy-повтор может не завершиться.

Язык грамматики:

```text
L(G) = { w | [[S]]_G(w, 0) = Ok |w| },
```

где `S` - стартовое правило.

## 5. Монадический парсер

В коде парсер задается типом:

```haskell
newtype Parser a = Parser
  { runParser :: RunState -> Either String (Maybe a, RunState) }
```

`RunState` хранит исходную строку, текущую позицию и memo-таблицу. `Maybe a` кодирует успех или отказ, `Either String` - аварийные ошибки, например обнаруженный во время выполнения циклический zero-width-вызов.

Реализованы экземпляры:

```haskell
Functor Parser
Applicative Parser
Monad Parser
Alternative Parser
```

Интуитивно:

- `pure x` - успешный парсер без потребления;
- `>>=` - последовательное применение парсеров;
- `empty` - отказ;
- `<|>` - PEG-выбор: сначала левая ветка, затем правая только при отказе левой, с откатом позиции.

Базовые комбинаторы:

```haskell
literal   :: String -> Parser ()
anyChar   :: Parser Char
charClass :: Bool -> [ClassAtom] -> Parser ()
look      :: Parser a -> Parser ()
neg       :: Parser a -> Parser ()
manyP     :: Parser () -> Parser ()
someP     :: Parser () -> Parser ()
optionalP :: Parser () -> Parser ()
named     :: String -> Parser () -> Parser ()
```

## 6. Компиляция AST в парсер

Компиляция задается структурной рекурсией по `Expr`:

```text
C[ε]          = pure ()
C["s"]        = literal s
C[.]          = anyChar *> pure ()
C[[...]]      = charClass ...
C[A]          = named "A" C[R(A)]
C[e1 ... ek]  = C[e1] *> ... *> C[ek]
C[e1 / ...]   = choose [C[e1], ...]
C[&e]         = look C[e]
C[!e]         = neg C[e]
C[e*]         = manyP C[e]
C[e+]         = someP C[e]
C[e?]         = optionalP C[e]
```

Проверка принадлежности запускает `C[S] *> endOfInput` из позиции `0`.

## 7. Packrat-мемоизация

Для каждого вызова нетерминала `A` в позиции `i` сохраняется результат:

```text
memo[(A, i)] = InProgress | Done Fail | Done (Ok j)
```

Алгоритм `named A p`:

1. Если `(A, i)` уже вычислено, вернуть сохраненный результат.
2. Если `(A, i)` помечено как `InProgress`, сообщить ошибку левой рекурсии или zero-width-цикла.
3. Иначе пометить `(A, i)` как `InProgress`, выполнить тело правила, затем сохранить успех или отказ.

Такой подход устраняет повторное вычисление одних и тех же рекурсивных вызовов и соответствует packrat-идее для PEG.

## 8. Валидация

Перед запуском выполняются проверки:

1. Нет дублирующихся правил.
2. Все нетерминалы, на которые есть ссылки, определены.
3. Стартовый нетерминал существует.
4. Нет левой рекурсии.
5. В `e*` и `e+` выражение `e` не может успешно завершиться без потребления входа.

Для пунктов 4-5 строится консервативная аппроксимация `zeroProgress(A)`: может ли правило `A` когда-либо завершиться успешно, не увеличив позицию. Затем строится граф первых zero-width-вызовов нетерминалов. Если нетерминал достижим из самого себя в этом графе, считается, что есть левая рекурсия.

## 9. Корректность: набросок доказательства

**Лемма 1.** Для любого выражения `e`, строки `w` и позиции `i`, если `C[e]` успешно завершается в позиции `j`, то `[[e]]_G(w, i) = Ok j`.

Доказательство - структурной индукцией по `e`. Для литералов, точки и классов утверждение следует из определения примитивных парсеров. Для последовательности используется индукционное предположение для первого и второго выражений. Для выбора используется тот факт, что `<|>` запускает правую ветку только после отказа левой и откатывает позицию. Для `&` и `!` используется то, что `look` и `neg` всегда восстанавливают исходную позицию.

**Лемма 2.** Если `[[e]]_G(w, i) = Ok j`, то `C[e]`, запущенный из позиции `i`, успешно завершается в позиции `j`.

Доказательство аналогично структурной индукцией. Для нетерминалов используется детерминированность PEG: результат правила в фиксированной позиции единственный, поэтому memo-таблица не меняет результат, а только переиспользует уже вычисленное значение.

**Теорема.** CLI печатает `ACCEPTED` тогда и только тогда, когда `w ∈ L(G)`.

Из лемм 1-2 для стартового выражения `S` и дополнительного парсера `endOfInput` следует, что успешное выполнение соответствует ровно случаю `[[S]]_G(w, 0) = Ok |w|`.

## 10. Сложность

Пусть `N` - число правил, `n` - длина входа. При отсутствии запрещенных zero-width-повторов и левой рекурсии каждая пара `(нетерминал, позиция)` вычисляется не более одного раза. Поэтому число вычислений правил ограничено `O(N * n)`. Память memo-таблицы также `O(N * n)`.

Стоимость терминальных проверок зависит от реализации строк. В теоретической оценке предполагается доступ к символу и проверка префикса за стоимость длины проверяемого литерала. В текущей реализации используется `String`, что удобно для учебного проекта; для больших входов его можно заменить на `Text` или массив символов.

## 11. Ограничения

- Не реализованы семантические действия и метки Peggy.
- Левые рекурсивные PEG-грамматики отвергаются.
- Валидация zero-width-ситуаций консервативна: некоторые безопасные грамматики с предикатами могут быть отвергнуты, зато интерпретатор не зациклится на greedy-повторе.
- Генератор Haskell-кода предполагает, что имена правил после санитарной замены не конфликтуют.

## 12. Вопросы к защите

**Почему выбор в PEG не такой, как в КС-грамматиках?**  
В PEG выбор упорядоченный: `e1 / e2` пробует `e2` только если `e1` отказало. Если `e1` успешно разобрало префикс, `e2` не рассматривается, даже если дальше весь разбор провалится.

**Зачем нужен `endOfInput`?**  
Без него парсер проверял бы только принадлежность некоторого префикса. Язык грамматики обычно определяется как множество строк, полностью разобранных стартовым правилом.

**Почему надо запрещать `e*`, если `e` может разобрать пустую строку?**  
Greedy-повтор будет бесконечно повторять успешный zero-width-разбор, не меняя позицию.

**Зачем memoization?**  
Без memoization рекурсивный backtracking может многократно пересчитывать один и тот же нетерминал на одной и той же позиции. Memoization делает такой вызов однократным.

**Чем проект связан с монадами?**  
Последовательность PEG выражается через `>>=`, отказ - через `empty`, упорядоченный выбор - через `<|>`, а весь разбор является композицией маленьких парсеров типа `Parser a`.

## 13. Воспроизведение сборки и проверки

Основная инструкция для запуска находится в `README.md`. Минимальная проверка:

```bash
cabal build
cabal test all
```

Полная проверка одной командой:

```bash
make check
```

Она собирает проект, запускает test-suite, проверяет CLI на примерах из
`examples/`, печатает нормализованную грамматику через `--ast`, генерирует
Haskell-модуль через `--emit-hs` и проверяет этот модуль командой
`ghc -fno-code`.

Ожидаемый результат test-suite:

```text
All tests passed
1 of 1 test suites (1 of 1 test cases) passed.
```
